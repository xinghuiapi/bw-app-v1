# 星汇项目版本发布与管理规范 (RELEASE.md)

为了保证 Android 和 iOS 端的版本一致性、上架合规性以及团队协作的顺畅，本项目统一遵循以下版本管理规则。

---

## 1. 唯一版本源：`pubspec.yaml`

所有版本号的修改**必须且仅在** `pubspec.yaml` 中进行。

- **格式**：`version: A.B.C+N`
- **映射关系**：
  - **`A.B.C`** (版本名) → Android `versionName` / iOS `CFBundleShortVersionString`
  - **`N`** (构建号) → Android `versionCode` / iOS `CFBundleVersion`

---

## 2. 版本名 (A.B.C) 升级策略 (SemVer 简化版)

- **A (主版本)**：重大改版、品牌重塑、不兼容的底层大改。
- **B (次版本)**：新增功能模块（如：新增钱包、新增游戏分类）。
- **C (修订号)**：修复 Bug、UI 小改、性能优化。

**示例**：`1.0.0` → `1.0.1` (修 Bug) → `1.1.0` (新功能) → `2.0.0` (大改版)。

---

## 3. 构建号 (N) 管理规则 (核心)

构建号 `N` 必须遵循 **“全局单调递增”** 原则，且 Android 与 iOS 同步使用。

- **递增时机**：每次准备发包给测试（APK/IPA）、提交到商店、或上传至 TestFlight 之前，必须将 `N` 加 1。
- **严禁重复**：同一个构建号 `N` 在 Git 历史中必须是唯一的。
- **好处**：避免商店上传冲突，确保覆盖安装逻辑正确。

---

## 4. 标准发版流程 (Release Workflow)

准备发布新版本时，请按以下步骤操作：

1. **同步代码**：拉取主分支最新代码，确保没有冲突。
2. **修改版本**：在 `pubspec.yaml` 中更新 `A.B.C`（如果需要）并将 `N` 递增。
3. **提交变更**：提交修改后的 `pubspec.yaml` 和 `pubspec.lock`。
4. **打 Tag**：在当前提交上打 Tag，格式为 `vA.B.C+N`。
   - 例如：`git tag v1.1.0+12`
5. **构建发布**：
   - Android: `flutter build apk --release`
   - iOS: `flutter build ipa --release`
6. **推送 Tag**：`git push origin --tags`

---

## 5. 特殊场景处理

### A. 紧急修复 (Hotfix)
- 增加修订号 `C`，例如 `1.1.0+12` 发现 Bug，修复后发布 `1.1.1+13`。

### B. 商店审核被拒 (Re-upload)
- **版本名 `A.B.C` 保持不变**。
- **构建号 `N` 继续递增**。
- 例如：`1.1.0+12` 被拒，修复后发布 `1.1.0+13`。

---

## 6. 团队协作约定

- **发包责任制**：谁负责本次发包，谁负责在主分支 bump `N`。
- **先同步后打包**：严禁在未拉取最新 `N` 的情况下私自打包发布，防止构建号冲突。
- **Git Status**：打包前确保工作区干净，所有改动已提交。

---

## 7. 版本示例

- 首个版本：`1.0.0+1`
- 第 10 次内部测试包：`1.0.0+10`
- 正式上架版本：`1.0.0+11`
- 新增功能后的首个包：`1.1.0+12`
- 修复 Bug：`1.1.1+13`
- 大版本更迭：`2.0.0+50`
